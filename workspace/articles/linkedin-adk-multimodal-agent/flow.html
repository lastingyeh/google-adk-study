Agentic Workflow Comparison

為什麼要選擇代理工作流程，而非傳統方法？

當需求不只是把流程跑完，而是還要能回報進度、解釋狀態、處理異常並理解使用者意圖時， 代理工作流程會比批次管道與事件導向架構更適合。

核心觀點

批次流程重視簡單，事件導向重視擴展，而代理工作流程則把 工具執行、狀態管理、錯誤處理與使用者互動 整合在同一條工作流中，讓流程不只自動化，更具備可理解與可調整能力。

📦批次資料管道

流程最簡單，但只要一個步驟失敗，整體流程就可能中斷。

01讀取檔案清單
→
02上傳至雲端儲存
→
03抽取內容
→
04寫入資料庫

主要問題

  • 檔案損毀或上傳失敗時，整體流程容易直接中斷。
  • 抽取結果異常時，缺乏彈性判斷與補救機制。
  • 即使結果為空，也可能繼續寫入資料庫造成浪費。
  • 使用者看不到進度，只能等待背景腳本執行結束。

⚡事件導向架構

具備雲端擴展能力，但服務分散、狀態難追、互動體驗較弱。

01上傳到 GCS
→
02事件觸發
→
03Cloud Function
→
04抽取 / 存檔

主要問題

  • 常需整合 GCS、Eventarc、Functions、Pub/Sub 等多個服務。
  • 狀態分散在不同事件與元件之間，追蹤不直觀。
  • 偵錯時需要跨服務查詢 log，維運成本較高。
  • 若要回報處理完成，還需額外設計輪詢或 WebSocket。

🤖代理工作流程

由代理統一協調流程，能理解情境、即時回報，並動態調整下一步。

01理解需求
→
02上傳檔案
→
03抽取資訊
→
04驗證 / 存檔 / 回報

核心優勢

  • 代理可在每一步即時說明目前進度與接下來要做的事。
  • 若結果異常，可主動停止、重試、跳過或請使用者補充資訊。
  • 流程狀態集中，不必在多個服務之間拼接上下文。
  • 可根據使用者意圖與情境，決定更合理的下一步。
「我先幫你上傳檔案」 「現在開始抽取內容」 「結果已成功保存」

三種方法比較

比較面向 批次資料管道 事件導向架構 代理工作流程 推薦
複雜度 低，通常一支腳本即可完成 高，需整合多個雲端服務 低到中，集中於單一代理流程
狀態管理 依賴程式內部變數 狀態分散、事件解耦 由代理統一理解與維護
錯誤處理 容易當機或直接跳出 多半只留在分散式 log 中 可即時回報並調整策略
使用者回饋 幾乎沒有 需額外補通知機制 回饋直接存在於對話中
適應能力 固定流程，彈性低 函式明確但僵硬 可依情境與意圖動態調整
情境感知 沒有 幾乎沒有 完整，可理解任務背景

結論整理

  • 批次流程適合簡單、固定、低風險任務，但缺乏韌性。
  • 事件導向架構適合高流量與雲端擴展，但工程與維運負擔較高。
  • 代理工作流程適合需要互動、解釋、判斷與應變的處理場景。
當需求不只是「把流程跑完」，而是還要「能回報、能判斷、能調整」時， 代理工作流程會是更好的選擇。
