## 場景 1: 首次使用 - GCS 路徑政策檢查

**用戶目標**: 第一次使用代理，檢查 GCS 儲存貯體中的資料是否符合 PII 政策。

### 對話流程

**👤 用戶**:
```
我想檢查我的資料是否包含個人識別資訊（PII）
```

**🤖 代理**:
```
好的！我可以幫您檢查 PII 政策違規。請問您想要針對以下哪個來源執行政策？
1. GCS 路徑（Google Cloud Storage）
2. Dataplex 搜尋查詢

請告訴我您的選擇。
```

**👤 用戶**:
```
GCS 路徑：gs://my-data-bucket/customer-data/
```

**🤖 代理**:
```
收到！讓我在記憶體中搜尋是否有類似的 PII 政策...

[呼叫 find_policy_in_memory 工具]

我沒有找到現有的 PII 政策。讓我為您生成一個新的政策程式碼...

[呼叫 generate_policy_code_from_gcs 工具]

✅ 已成功生成 PII 檢查政策！該政策將檢查以下欄位：
- Email 地址
- 電話號碼
- 社會安全號碼
- 信用卡號碼

我已將此政策儲存到記憶體（版本 1）。現在開始執行政策檢查...

[呼叫 run_policy_from_gcs 工具]

📊 政策執行結果：
發現 3 個違規項目：
1. customer_info.csv - 包含未加密的 email 欄位
2. transactions.parquet - 包含信用卡號碼
3. users.json - 包含電話號碼

您需要修復建議嗎？
```

**👤 用戶**:
```
是的，請提供修復建議
```

**🤖 代理**:
```
[呼叫 suggest_remediation 工具]

🔧 修復建議：

1. **customer_info.csv**
   - 問題：未加密的 email 欄位
   - 建議：使用 Cloud KMS 加密或雜湊處理該欄位
   - 程式碼範例：
     ```python
     from google.cloud import kms
     # 使用 KMS 加密 email 欄位
     ```

2. **transactions.parquet**
   - 問題：包含信用卡號碼
   - 建議：移除或遮罩信用卡號碼，僅保留後 4 位數

3. **users.json**
   - 問題：包含電話號碼
   - 建議：對電話號碼進行雜湊或遮罩處理
```

```

```

**🤖 代理**:
```
請問您想要針對哪個來源執行政策？
1. GCS 路徑

```

**👤 用戶**:
```

```

**🤖 代理**:
```
[呼叫 find_policy_in_memory 工具]
[呼叫 generate_policy_code_from_gcs 工具]
[呼叫 save_policy_to_memory 工具]
[呼叫 run_policy_from_gcs 工具]

📊 安全性檢查結果：
發現 7 個安全問題：
1. config.json - 包含明文密碼
2. api_keys.txt - API 金鑰未加密
3. secrets/ - 資料夾權限過於寬鬆
4. db_backup.sql - 包含敏感資料且未加密
5. credentials.yaml - IAM 憑證洩漏風險
6. logs/ - 日誌包含個人資訊
7. temp/ - 暫存資料未清理

正在生成修復建議...

[呼叫 suggest_remediation 工具]

🔧 詳細修復建議：

1. **config.json - 明文密碼**
   - 嚴重程度：🔴 高
   - 建議：使用 Secret Manager 儲存密碼
   - 步驟：
     1. 建立 Secret Manager 資源
     2. 遷移密碼到 Secret Manager
     3. 更新應用程式讀取 Secret
   - 範例程式碼：
     ```python
     from google.cloud import secretmanager
     client = secretmanager.SecretManagerServiceClient()
     name = "projects/PROJECT_ID/secrets/SECRET_ID/versions/latest"
     response = client.access_secret_version(name=name)
     password = response.payload.data.decode('UTF-8')
     ```

2. **api_keys.txt - API 金鑰未加密**
   - 嚴重程度：🔴 高
   - 建議：使用 Cloud KMS 加密或遷移至 Secret Manager

[... 更多建議 ...]

您想匯出這個報告嗎？
```

```
匯出為 CSV 格式到 gs://security-reports/
```

**🤖 代理**:
```
[呼叫 export_report 工具，格式為 CSV]

- 包含：違規項目、嚴重程度、修復建議

## 最佳實踐建議

### 🎯 提示詞撰寫技巧

1. **明確指定來源類型**
   - ✅ 好：「檢查 GCS 路徑 gs://my-bucket/ 的 PII」
   - ❌ 差：「檢查我的資料」

2. **使用具體的政策描述**
   - ✅ 好：「檢查欄位命名是否符合 snake_case 規範」
   - ❌ 差：「檢查格式」

3. **善用過濾條件**
   - 提供 author、日期範圍可以更精確找到政策

4. **定期記憶體維護**
   - 每月執行一次 prune_memory
   - 對常用政策進行評分

### 🔄 工作流程建議

1. **首次使用新政策**：
   - 搜尋記憶體 → 未找到 → 生成 → 儲存 → 執行

2. **重複使用**：
   - 搜尋記憶體 → 找到 → 確認版本 → 執行

3. **政策演進**：
   - 找到舊版本 → 提供更新需求 → 生成新版本 → 執行 → 評分

4. **合規監控**：
   - 定期執行計分卡 → 分析趨勢 → 優先修復 → 匯出報告

## 常見問題排解

### Q: 找不到我之前建立的政策？
A: 嘗試：
- 使用更通用的查詢詞
- 不指定過濾條件（author、日期）
- 詢問代理「列出我最近建立的政策」

### Q: 政策版本太多，如何清理？
A: 使用 `prune_memory` 工具，可以設定：
- 保留最近 N 天的政策
- 保留使用次數 > N 的政策
- 保留評分 > N 的政策

### Q: 如何匯出報告到本地？
A: 提供 GCS 路徑，報告上傳後可從 Cloud Console 下載。或要求匯出為 CSV/HTML 格式方便本地開啟。

### Q: 執行歷史可以保留多久？
A: 預設保留 90 天，可通過配置調整。
